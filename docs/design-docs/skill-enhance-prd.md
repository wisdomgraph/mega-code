## 1. Executive Summary

The Skill Enhancement System is a Claude Code skill (`/mega-code:skill-enhance`) that measures, reviews, and iteratively improves mega-code skills using an LLM-as-judge A/B testing loop. Given a target skill, it generates realistic test cases, runs isolated agent completions with and without the skill as a system prompt, grades both outputs against per-expectation criteria, presents results in a browser-based HTML review UI, and produces an enhanced version of the skill ŌĆö all within a single interactive session. A static security audit gates A/B execution, a configurable trust policy drives the proceed/warn/skip decision, and benchmark data is persisted to an iteration workspace for longitudinal comparison. The complete cycle can repeat across multiple iterations until the user is satisfied with the enhancement.

---

## 2. Problem Statement

Skills generated by the MEGA-Code pipeline are delivered to developers without structured quality assurance. There is no mechanism to:

1. **Measure** whether a skill actually improves LLM output quality versus an unprimed baseline ŌĆö skills are accepted on trust rather than evidence.
2. **Detect regressions** introduced by skill updates; each version is shipped without comparison to the prior version.
3. **Surface security red flags** before running untrusted skill content as a system prompt in isolated agent subprocesses.
4. **Collect structured human feedback** on individual test cases at the expectation level, which is needed to guide focused improvement.
5. **Iteratively refine** a skill in response to eval findings ŌĆö there is no controlled loop that re-evaluates an enhanced skill against the same test suite.

The result is that skill quality is opaque. Developers have no confidence score, no A/B evidence, and no systematic path from "skill generated" to "skill verified and improved".

## 3. Goals and Non-Goals

### Goals

- **G1**: Provide a single-command entry point (`/mega-code:skill-enhance`) that evaluates any mega-code skill and produces an enhanced version.
- **G2**: Automate A/B test execution ŌĆö isolated agent CLI subprocesses produce with-skill and baseline completions for every test case with no manual intervention.
- **G3**: Enforce a static security gate (7-category automated scan with 10 regex rules + 2 manual-review categories, trust classification, `full_access` / `warn_and_continue` / `skip_ab` policy) before any A/B execution.
- **G4**: Grade outputs with strict LLM-as-judge evaluation, producing per-expectation pass/fail verdicts with supporting evidence.
- **G5**: Present results in a browser-based HTML review UI (localhost:3117) that collects per-test-case free-text feedback.
- **G6**: Produce an enhanced skill by applying four documented improvement principles informed by eval findings and user feedback.
- **G7**: Persist benchmark data (performance increase, token savings, BENEFICIAL/NOT_BENEFICIAL verdict) to a structured iteration workspace and store the enhanced skill version on the remote server.

### Non-Goals

- **NG1**: Server-side skill generation and pipeline orchestration ŌĆö this PRD covers the client-side evaluation and enhancement loop only.
- **NG2**: Real-time continuous monitoring of deployed skills ŌĆö the system runs on-demand at the user's request.
- **NG3**: Multi-model comparison ŌĆö all A/B completions use the same agent CLI (claude or codex) that the host is already running; model selection is not exposed.
- **NG4**: Automatic deployment of enhanced skills without user confirmation ŌĆö the user explicitly controls whether to accept and publish the enhanced version.

## 5. Functional Requirements

### FR-0: Authentication Gate

- **FR-0.1**: Before any workflow phase, run `uv run --directory "$MEGA_DIR" python -m mega_code.client.check_auth` to verify the API key is valid and not expired.
- **FR-0.2**: If the auth check exits non-zero, display the output to the user and halt the workflow entirely.
- **FR-0.3**: Auth failure messages follow the same branching as the wisdom-curate system: expired key ŌåÆ direct to `/mega-code:login`; missing key ŌåÆ direct to `mega-code configure --api-key`; `ConnectError`/`TimeoutException` ŌåÆ display connection error.

### FR-1: Skill Selection & Discovery

- **FR-1.1**: If a skill name is provided as `$ARGUMENTS`, use it directly without author filtering. Direct arguments are not restricted to mega-code-authored skills.
- **FR-1.2**: If no argument is provided, run `list-skills` to enumerate available mega-code skills and present them to the user via `AskUserQuestion`.
- **FR-1.3**: `list_skills()` scans four sources in priority order, deduplicating by name: project-scoped installed skills (`{project_dir}/.claude/skills/`), user-scoped installed skills (`~/.claude/skills/`), pending skills (`PENDING_SKILLS_DIR`), archived runs (`FEEDBACK_DIR/{project_id}`).
- **FR-1.4**: Author filtering uses `MEGACODE_AUTHOR_MARKER` (case-insensitive) checked against the `author` field (top-level or `metadata.author`).
- **FR-1.5**: Project directory detection applies a three-step resolution: session-backed project mapping via `CLAUDE_SESSION_ID`, then `CLAUDE_PROJECT_DIR` env var, then `cwd` ŌĆö skipping any path under `~/.claude/plugins/cache/` or `~/.claude/plugins/marketplaces/`.
- **FR-1.6**: If exactly one skill is returned by `list-skills`, skip `AskUserQuestion` and proceed directly with that skill.
- **FR-1.7**: Once a skill is selected, run `resolve-skill --name "$SKILL_NAME"` immediately. This repairs the skill directory name to the canonical slug before any subsequent phases.
- **FR-1.8**: `resolve-skill` outputs a single-line JSON object: `{"name": "...", "path": "...", "content": "..."}`. The orchestrator captures `SKILL_NAME`, `SKILL_PATH`, and skill content from this output.

### FR-2: Workspace Management

- **FR-2.1**: Run `eval_workspace create-iteration --skill-name "$SKILL_NAME" --skill-path "$SKILL_PATH"` to create the next iteration directory.
- **FR-2.2**: The workspace root is `{data_dir()}/data/skill-enhance/{canonical_skill_name}/`. Each iteration lives at `iteration-N/` where N is the next integer after the current maximum.
- **FR-2.3**: `create-iteration` outputs a single JSON line: `{"path": "/abs/path/to/iteration-N", "iteration": N}`.
- **FR-2.4**: Path traversal is rejected at two levels: `_validate_path_component()` rejects skill names containing `..`, `/`, or null bytes; `save_artifact()` and `save_text_artifact()` verify that the resolved artifact path is relative to the iteration directory.
- **FR-2.5**: All structured artifacts are written to the iteration directory: `test-cases.json`, `security-review.json`, `ab-results.json`, `gradings.json`, `eval-full.json`, `benchmark.json`, `feedback.json`, `enhanced-skill.md`, `original-skill.md`, `skill-identity.json`, and `viewer.log` (stderr from the viewer subprocess).

### FR-3: Test Case Generation

- **FR-3.1**: The host agent (LLM) reads the resolved skill content and generates exactly 4 test cases following the schema in `references/test-case-schema.md`.
- **FR-3.2**: Each test case has a `task` field (a realistic coding request) and an `expectations` list of 3ŌĆō4 specific, verifiable natural-language assertions.
- **FR-3.3**: Expectations must distinguish responses that follow the skill from those that do not. Vague quality claims are not acceptable.
- **FR-3.4**: Test cases are written to `$ITER_DIR/test-cases.json` in `EvalTestSuite` format: `{"cases": [...]}`.
- **FR-3.5**: The runner also accepts a bare list (without the `cases` wrapper) for backward compatibility.

### FR-4: Security Audit

- **FR-4.1**: Run `python -m mega_code.client.skill_security_audit --skill-path "$SKILL_PATH" --iteration-dir "$ITER_DIR"` before any A/B execution.
- **FR-4.2**: The scanner applies 10 regex rules across 7 automated categories: Credential & Secret Management, Command Injection, Prompt Injection & Instruction Override, Data Exfiltration, Supply Chain & Dependencies, Dual-Layer Attack Detection, Hook & Config Exploitation. Two additional categories ŌĆö Principle of Least Privilege and Indirect Prompt Injection via Content Processing ŌĆö are manual-review categories assessed by the host agent using the checklist in `references/security-checklist.md`, not by automated pattern matching.
- **FR-4.3**: Trust classification: skills with `MEGACODE_AUTHOR_MARKER` in the `author` field are `trusted`; all others are `semitrusted`. Explicit `source` overrides via `--source` CLI flag are supported.
- **FR-4.4**: Security score is a discrete integer from the set {1, 3, 4, 5} (score 2 is not used): 5 = no findings; 4 = low-severity findings only; 3 = medium severity or 3+ findings; 1 = any high-severity finding.
- **FR-4.5**: A/B policy derivation: `skip_ab` if high-severity findings and `semitrusted`; `warn_and_continue` if any findings (including for `trusted`); `full_access` only if no findings.
- **FR-4.6**: The scanner writes `security-review.json` to the iteration directory. The host agent reads both `security-review.json` and the raw skill content to explain findings to the user.
- **FR-4.7**: `security_review.ignore_patterns` is advisory only. The auditor applies a safe allowlist (`_ALLOWED_IGNORE_PATTERNS = {"env_reference"}`). High-risk findings cannot be suppressed.
- **FR-4.8**: For `semitrusted` skills, the orchestrator always asks the user whether to proceed before Phase 4, even if `ab_policy` is `full_access`.
- **FR-4.9**: If the user chooses static-only review after `skip_ab`, the orchestrator summarizes findings and stops.

### FR-5: A/B Test Execution

- **FR-5.1**: Run `python -m mega_code.client.skill_enhance_runner --test-cases "$ITER_DIR/test-cases.json" --skill-md "$SKILL_PATH" --agent "${EVAL_AGENT}" --output "$ITER_DIR/ab-results.json"`. If `EVAL_AGENT` is empty, omit `--agent`.
- **FR-5.2**: For each test case, two isolated completions are spawned: the with-skill arm passes SKILL.md content as `system_prompt`; the baseline arm uses a minimal instruction.
- **FR-5.3**: Completions run via `host_llm.complete()` which invokes the agent CLI as a subprocess with tools disabled and plugin environment variables stripped.
- **FR-5.4**: All test-case pairs run concurrently via `asyncio.gather()`, bounded by a semaphore of size `MEGA_CODE_EVAL_CONCURRENCY` (default: 4).
- **FR-5.5**: Each completion is subject to `MEGA_CODE_EVAL_TIMEOUT` seconds (default: 300s). On timeout, the process is killed.
- **FR-5.6**: Each output dict contains: `task`, `expectations`, `with_skill_output`, `baseline_output`, `with_skill_model`, `baseline_model`, `with_skill_tokens`, `baseline_tokens`.

### FR-6: LLM Grading

- **FR-6.1**: The host agent reads `$ITER_DIR/ab-results.json` and grades both outputs for every test case following the schema in `references/grading-schema.md`.
- **FR-6.2**: Grading is strict: an expectation passes only if it is clearly and unambiguously met.
- **FR-6.3**: Each grading entry has three fields: `expectation` (text), `passed` (bool), `evidence` (one-sentence quote).
- **FR-6.4**: Gradings are written to `$ITER_DIR/gradings.json` as a list of objects per test case, each containing `with_skill_gradings` and `baseline_gradings`.
- **FR-6.5**: Extra gradings beyond the expectation count (LLM over-generation) are truncated in `aggregate_eval_result()`.

### FR-7: Results Aggregation

- **FR-7.1**: The orchestrator assembles `$ITER_DIR/eval-full.json` combining `skill_name`, `model`, `test_cases`, `ab_outputs`, and `gradings`.
- **FR-7.2**: Run `python -m mega_code.client.skill_enhance_aggregator --eval-data "$ITER_DIR/eval-full.json" --iteration-dir "$ITER_DIR"` to produce `benchmark.json`.
- **FR-7.3**: Per-test scores: `with_skill_score = passed_count / expectation_count`; `baseline_score = passed_count / expectation_count`.
- **FR-7.4**: Aggregate metrics: `performance_increase = with_skill_avg - baseline_avg`; `token_savings = 1 - (with_total_tokens / baseline_total_tokens)`.
- **FR-7.5**: Token counts come from runner output. If not present, text length is used as a fallback proxy.
- **FR-7.6**: Verdict: `BENEFICIAL` if `performance_increase > MEGA_CODE_BEHAVIORAL_LIFT_THRESHOLD` (default 5%) OR if `performance_increase >= 0` AND `token_savings > MEGA_CODE_BEHAVIORAL_SAVINGS_THRESHOLD` (default 20%). Otherwise `NOT_BENEFICIAL`.
- **FR-7.7**: The aggregator prints a markdown summary table plus ROI metrics to stdout for user display.

### FR-8: HTML Review & Feedback

- **FR-8.1**: Launch the review viewer using `launch-viewer.sh "$MEGA_DIR" "$ITER_DIR" "$SKILL_NAME" "$ITERATION_NUM"`.
- **FR-8.2**: The launch script kills any stale process on port 3117 via `lsof -ti :3117 | xargs kill`, starts the viewer as a background process with `--no-browser` (so the Python process binds the port without blocking on browser launch), polls `http://localhost:3117` for readiness (up to 10 seconds, checked every 0.5s), then opens the browser itself via `xdg-open` (Linux) or `open` (macOS). The `--no-browser` flag exists precisely because prior versions of `enhancement_viewer.py` called `webbrowser.open()` before `serve_forever()`, which could block indefinitely in terminal-only environments and leave the port unbound.
- **FR-8.3**: The viewer serves a self-contained HTML page (no external CDN dependencies) displaying: skill name, iteration number, performance metrics, per-test-case A/B comparison with pass/fail badges, and a feedback textarea per test case.
- **FR-8.4**: If `ITERATION_NUM > 1`, the viewer is launched with `--previous-workspace` for side-by-side comparison with the prior iteration.
- **FR-8.5**: The HTTP server binds to `127.0.0.1` only. POST body size is capped at 5 MB.
- **FR-8.6**: `GET /api/feedback` returns `feedback.json` contents (or `{}` if not yet written).
- **FR-8.7**: `POST /api/feedback` validates JSON with a `reviews` key, writes to `$ITER_DIR/feedback.json`, returns `{"ok": true}`.
- **FR-8.8**: The orchestrator waits for user confirmation before stopping the viewer via `stop-viewer.sh`.
- **FR-8.9**: If the user has no feedback, the orchestrator proceeds with empty feedback.

### FR-9: Skill Enhancement

- **FR-9.1**: The host agent reads: `$ITER_DIR/eval-full.json`, `$ITER_DIR/feedback.json`, and the current `SKILL_PATH`.
- **FR-9.2**: Enhancement follows the four principles from `references/enhancement-principles.md`:
  1. **Generalize from feedback** ŌĆö avoid overfitting to specific test cases.
  2. **Keep the prompt lean** ŌĆö remove instructions that are not helping.
  3. **Explain the why** ŌĆö avoid `ALWAYS`/`NEVER` in caps; explain reasoning.
  4. **Look for repeated work** ŌĆö bundle common scripts into `scripts/`.
- **FR-9.3**: The enhanced skill must satisfy a frontmatter checklist: (1) `---`/`---` YAML block present; (2) `name` and `description` present; (3) `metadata.tags` as a list with at least 2 entries; (4) skill body incorporates eval findings and feedback.
- **FR-9.4**: The enhanced skill is written to `$ITER_DIR/enhanced-skill.md`.
- **FR-9.5**: After writing, the orchestrator verifies `metadata.tags` is present. If missing, the file is edited before proceeding.

### FR-10: Accept & Publish

- **FR-10.1**: Run `skill_enhance_helper accept-skill` with `--skill-path`, `--enhanced-path`, `--iteration-dir`, `--iteration`, and `--benchmark`.
- **FR-10.2**: `accept_enhanced_skill()` performs: (1) backup original to `$ITER_DIR/original-skill.md`; (2) bump semantic minor version; (3) refresh `generated_at` timestamp; (4) inject ROI from benchmark; (5) rename directory to canonical slug if needed; (6) write enhanced content over original `SKILL_PATH`.
- **FR-10.3**: `skill-identity.json` records `original_skill_name`, `canonical_skill_name`, and `original_skill_path` for server upload.
- **FR-10.4**: ROI injection uses `format_eval_roi_entry()` producing: `model`, `performance_increase`, `token_savings`, `test_count`, `with_success_rate`, `baseline_success_rate`.
- **FR-10.5**: Run `skill_enhance_helper store-skill` for remote server storage. Server storage is skipped if `MEGA_CODE_CLIENT_MODE != "remote"`.
- **FR-10.6**: On store failure, the orchestrator warns the user that the skill was saved locally but not stored on the server.

### FR-11: Iteration Loop

- **FR-11.1**: After Phase 8, the orchestrator asks: "Would you like to run another iteration to validate the enhancement?"
- **FR-11.2**: If yes, run `eval_workspace create-iteration` again and return to Phase 2. The skill being evaluated is now the in-place enhanced version.
- **FR-11.3**: If no, display a summary of actions and file locations.

## 6. Non-Functional Requirements

### NFR-1: Security

- **NFR-1.1**: The security scanner applies 10 regex patterns across 7 automated categories (plus 2 manual-review categories assessed by the host agent). Only `env_reference` is suppressible via `_ALLOWED_IGNORE_PATTERNS`. High-severity findings cannot be suppressed.
- **NFR-1.2**: Trust policy is enforced strictly: `semitrusted` + high-severity ŌåÆ `skip_ab`; any findings (even trusted) ŌåÆ `warn_and_continue`; no findings ŌåÆ `full_access`.
- **NFR-1.3**: Path traversal is blocked at two enforcement points: `_validate_path_component()` and `path.resolve().is_relative_to()` checks in artifact save functions.
- **NFR-1.4**: `accept_enhanced_skill()` includes a path guard on `metadata.json`: only updates under `PENDING_SKILLS_DIR` or `FEEDBACK_DIR`.
- **NFR-1.5**: The review HTTP server binds to `127.0.0.1` only. POST body capped at 5 MB. HTML output is XSS-escaped.
- **NFR-1.6**: Agent subprocesses run with a sanitized environment that strips `CLAUDE_PLUGIN_ROOT`, `CLAUDE_PROJECT_DIR`, and `MEGA_CODE_CLIENT_MODE`.

### NFR-2: Reliability

- **NFR-2.1**: Per-completion timeout of `MEGA_CODE_EVAL_TIMEOUT` (default 300s). On timeout, `proc.kill()` is called.
- **NFR-2.2**: Concurrency bounded by `asyncio.Semaphore` of size `MEGA_CODE_EVAL_CONCURRENCY` (default 4).
- **NFR-2.3**: A/B runner failure causes the orchestrator to show the error and stop ŌĆö no partial results accepted. `TimeoutError` (from `MEGA_CODE_EVAL_TIMEOUT`) and `RuntimeError` (from non-zero agent CLI exit) are distinct failure modes: timeout suggests increasing the timeout; non-zero exit suggests an agent configuration problem.
- **NFR-2.4**: Extra gradings beyond expectation count are truncated rather than causing validation errors.
- **NFR-2.5**: Edge case handling: (a) Empty test cases (`cases: []`) ŌĆö the A/B runner produces empty `ab-results.json`, and the aggregator exits with code 1 ("No test cases found"). (b) Zero expectations in a test case ŌĆö `aggregate_eval_result()` returns `0.0` score for both arms via `if total_exp else 0.0` guard. (c) All gradings fail ŌĆö produces `with_skill_avg = 0.0`, `baseline_avg = 0.0`, `performance_increase = 0.0`, verdict `NOT_BENEFICIAL`. (d) `benchmark.json` absent when `store-skill` runs without `--benchmark` ŌĆö ROI is not injected, skill stored without performance metadata.
- **NFR-2.6**: Viewer startup failure ŌĆö if `launch-viewer.sh` cannot bind port 3117 within 10 seconds, it exits non-zero and prints a warning directing the user to `$ITER_DIR/viewer.log`. The orchestrator should check the exit status and inform the user of the failure path.

### NFR-3: Observability

- **NFR-3.1**: `skill_enhance_helper` operations are instrumented with OpenTelemetry spans via `@traced()`. Tracing is optional ŌĆö degrades gracefully when package is absent.
- **NFR-3.2**: Span attributes include: `skill_enhance.skill_count`, `skill_enhance.skill_names`, `skill_enhance.version`, `skill_enhance.iteration`.
- **NFR-3.3**: `benchmark.json` is the durable performance record per iteration.

### NFR-4: Data Integrity

- **NFR-4.1**: Each iteration produces an isolated, self-contained directory. Artifacts from one iteration do not overwrite artifacts from another.
- **NFR-4.2**: Skill name validation rejects empty strings, `..` sequences, `/` characters, and null bytes.
- **NFR-4.3**: `skill-identity.json` decouples the original pending-skill folder name from the canonical slug for correct server storage.

### NFR-5: Portability

- **NFR-5.1**: XDG Base Directory Specification compliant, overridable via `MEGA_CODE_DATA_DIR`.
- **NFR-5.2**: Agent CLI detection supports both `claude` and `codex` via `shutil.which()`.
- **NFR-5.3**: Python 3.11ŌĆō3.13 compatibility.
- **NFR-5.4**: The HTML review UI has no external CDN dependencies.

### NFR-6: Performance

- **NFR-6.1**: A/B completions (8 total invocations for the default 4 test cases ├Ś 2 arms) run concurrently via `asyncio.gather()`. Wall-clock time is dominated by the slowest individual completion.
- **NFR-6.2**: `detect_agent_cli()` and `_read_codex_model()` are cached for the process lifetime.
- **NFR-6.3**: The review viewer regenerates HTML on each `GET /` request (hot reload from `eval-full.json`).

## 7. Data Models

### 7.1 EvalExpectation

| Field | Type | Description |
|-------|------|-------------|
| `text` | `str` | Natural-language assertion about expected output quality |

### 7.2 EvalTestCase

| Field | Type | Description |
|-------|------|-------------|
| `task` | `str` | Realistic coding task prompt written as a real user would |
| `expectations` | `list[EvalExpectation]` | 3ŌĆō4 specific, verifiable assertions |

### 7.3 EvalTestSuite

| Field | Type | Description |
|-------|------|-------------|
| `cases` | `list[EvalTestCase]` | Container for all generated test cases (exactly 4 by convention) |

### 7.4 EvalGrading

| Field | Type | Description |
|-------|------|-------------|
| `expectation` | `str` | The expectation text being graded |
| `passed` | `bool` | True only if the expectation is clearly and unambiguously met |
| `evidence` | `str` | One-sentence evidence quote from the output |

### 7.5 EvalGradingResult

| Field | Type | Description |
|-------|------|-------------|
| `gradings` | `list[EvalGrading]` | Per-expectation grading results for a single output arm |

> **Note**: `EvalGradingResult` is a Pydantic model representing one arm of one test case. The `gradings.json` file uses a different wire format: a list of objects with `with_skill_gradings` and `baseline_gradings` keys (matching the `EvalTestResult` structure), not `EvalGradingResult` directly. The host agent writes this wire format; `aggregate_eval_result()` consumes it.

### 7.6 EvalTestResult

| Field | Type | Description |
|-------|------|-------------|
| `test_case` | `EvalTestCase` | The test case being graded |
| `with_skill_gradings` | `list[EvalGrading]` | Grading results for the with-skill completion |
| `baseline_gradings` | `list[EvalGrading]` | Grading results for the baseline completion |
| `with_skill_score` | `float` | Fraction of expectations met by with-skill arm (0.0ŌĆō1.0) |
| `baseline_score` | `float` | Fraction of expectations met by baseline arm (0.0ŌĆō1.0) |
| `with_skill_tokens` | `int` | Output tokens from with-skill completion (fallback: text length) |
| `baseline_tokens` | `int` | Output tokens from baseline completion (fallback: text length) |

### 7.7 EvalResult

| Field | Type | Description |
|-------|------|-------------|
| `skill_name` | `str` | Name of the evaluated skill |
| `model` | `str` | Model identifier from agent CLI output (or `"host-agent"`) |
| `test_results` | `list[EvalTestResult]` | Per-test-case aggregated results |
| `with_skill_avg` | `float` | Mean with-skill score across all test cases |
| `baseline_avg` | `float` | Mean baseline score across all test cases |
| `performance_increase` | `float` | `with_skill_avg - baseline_avg` |
| `token_savings` | `float` | `1 - (with_total_tokens / baseline_total_tokens)` |
| `with_skill_total_tokens` | `int` | Sum of with-skill tokens across all test cases |
| `baseline_total_tokens` | `int` | Sum of baseline tokens across all test cases |
| `verdict` | `Literal["BENEFICIAL", "NOT_BENEFICIAL"]` | Computed by `compute_verdict()` |

### 7.8 CompletionResult (from host_llm.py)

| Field | Type | Description |
|-------|------|-------------|
| `text` | `str` | Full text output from the agent CLI completion |
| `model` | `str` | Model identifier reported by the agent (e.g., `claude-sonnet-4-6`) |
| `cost_usd` | `float` | Estimated cost in USD (0.0 if unavailable) |
| `duration_ms` | `int` | Wall-clock execution time in milliseconds |
| `is_error` | `bool` | True if the completion ended in an error state |
| `output_tokens` | `int` | Output token count from `usage.output_tokens` (claude) or `turn.completed` event (codex). Defaults to 0 when unavailable ŌĆö `aggregate_eval_result()` then falls back to `len(output_text)` as a proxy. |

> **Design Note**: Token count extraction differs by agent CLI. Claude reports `usage.output_tokens` in its JSON envelope. Codex reports tokens via JSONL `turn.completed` events. When neither is available, the fallback to text length inflates or deflates `token_savings` ŌĆö see Known Limitation #5.

## 8. API Contracts

### 8.1 Internal CLI Contracts

Each Python module exposes a `__main__` entry point invoked via `uv run --directory "$MEGA_DIR" python -m <module>`.

#### `mega_code.client.skill_enhance_helper`

**Subcommand: `list-skills`**

```
python -m mega_code.client.skill_enhance_helper list-skills [--project-dir <path>]

Output (stdout): JSON array
[{"name": "...", "description": "...", "state": "pending|installed|archived", "path": "..."}]
```

**Subcommand: `resolve-skill`**

```
python -m mega_code.client.skill_enhance_helper resolve-skill --name <skill-name> [--project-dir <path>]

Output (stdout): single-line JSON
{"name": "...", "path": "...", "content": "..."}

Exit codes: 0 = success, 1 = not found or ambiguous
```

**Subcommand: `accept-skill`**

```
python -m mega_code.client.skill_enhance_helper accept-skill \
    --skill-path <path> --enhanced-path <path> --iteration-dir <path> \
    --iteration <int> [--benchmark <path>]

Output (stdout): absolute path of the final replaced SKILL.md
Side effects: writes original-skill.md, skill-identity.json, bumps version, injects ROI
```

**Subcommand: `store-skill`**

```
python -m mega_code.client.skill_enhance_helper store-skill \
    --skill-name <str> --iteration-dir <path> --iteration <int> \
    --skill-path <path> [--benchmark <path>]

Exit codes: 0 = success, 1 = unrecoverable error
```

#### `mega_code.client.eval_workspace`

```
python -m mega_code.client.eval_workspace create-iteration \
    --skill-name <str> [--skill-path <path>]

Output (stdout): {"path": "/abs/path/to/iteration-N", "iteration": N}
```

#### `mega_code.client.skill_security_audit`

```
python -m mega_code.client.skill_security_audit \
    --skill-path <path> [--iteration-dir <path>] \
    [--source auto|same-repo|known-org|unknown] [--debug]
```

## 11. Security Considerations

### 11.1 Threat Model

| Threat | Mitigation | Implementation |
|--------|-----------|----------------|
| **Command injection via skill content** | Static pattern scan; `curl_pipe_bash`, `wget_pipe_shell`, `eval_curl`, `shell_interpolation` patterns flagged | `skill_security_audit.py`: 10 compiled regexes; `skip_ab` halts execution for high-severity + semitrusted |
| **Prompt injection** | `prompt_override` pattern detects instruction override language; skill content only passed to isolated subprocesses | `--disable-slash-commands`, `--setting-sources ""` flags strip plugin context |
| **Data exfiltration** | `external_post` pattern detects outbound curl/wget with data upload flags | `derive_ab_policy()` enforces `skip_ab` for high-severity + semitrusted |
| **Supply chain attacks** | `curl_pipe_bash` and `wget_pipe_shell` patterns; high severity ensures `skip_ab` for semitrusted | Separate rules per tool, both HIGH severity |
| **Credential exposure** | `credential_paths` detects `~/.ssh`, `~/.aws/credentials`, `id_rsa` references | HIGH severity pattern |
| **XSS in viewer** | JSON serialization only; no user-controlled raw HTML rendering | `json.dumps()` with proper escaping |
| **Path traversal** | `_validate_path_component()` + `is_relative_to()` checks | `eval_workspace.py` enforcement at two levels |
| **Trust escalation** | Trust is provenance-only; does not suppress findings. Trusted + high = `warn_and_continue` | `derive_ab_policy()`: `full_access` only when no findings at all |
| **Ignore pattern abuse** | Only `env_reference` is suppressible; high-severity patterns never suppressible | `_ALLOWED_IGNORE_PATTERNS` intersection |
| **Subprocess env leakage** | `CLAUDE_PLUGIN_ROOT`, `CLAUDE_PROJECT_DIR`, `MEGA_CODE_CLIENT_MODE` stripped | `host_llm.py`: `_clean_env()` |

### 11.2 Security Pattern Reference

| Pattern Name | Category | Severity | Detects |
|---|---|---|---|
| `curl_pipe_bash` | Supply Chain & Dependencies | HIGH | `curl ... \| bash/sh` |
| `wget_pipe_shell` | Supply Chain & Dependencies | HIGH | `wget ... \| bash/sh` |
| `eval_curl` | Command Injection | HIGH | `eval $(curl ...)` |
| `credential_paths` | Credential & Secret Management | HIGH | `~/.ssh`, `~/.aws/credentials`, `id_rsa` |
| `prompt_override` | Prompt Injection | HIGH | "ignore instructions", "bypass safety" |
| `config_persistence` | Hook & Config Exploitation | HIGH | Writing `.claude/settings.json`, hooks |
| `obfuscated_payload` | Dual-Layer Attack Detection | MEDIUM | base64/atob/decode, long hex strings |
| `external_post` | Data Exfiltration | MEDIUM | curl/wget with `--data`, `-d`, `--upload-file` |
| `shell_interpolation` | Command Injection | MEDIUM | `bash -c`, `subprocess.run`, `exec` |
| `env_reference` | Credential & Secret Management | LOW | `.env` file reference (suppressible) |

## 14. Testing Strategy

### 14.1 Unit Tests

- **Model computation**: `compute_verdict()` boundary conditions at threshold values; `aggregate_eval_result()` with mismatched grading lengths (truncation); token estimation fallback; `EvalResult` serialization round-trips.
- **Security patterns**: Each of 10 `_PATTERN_RULES` with positive and negative examples; `security_score()` mapping; `derive_ab_policy()` exhaustive truth table; `_applied_ignored_pattern_names()` allowlist enforcement; trust classification with author marker variants.
- **Workspace management**: `workspace_root()` path traversal rejection; `create_iteration_dir()` auto-increment; `save_artifact()` path escape prevention; `_validate_path_component()` edge cases.
- **Skill helper**: `bump_minor_version()` edge cases; `accept_enhanced_skill()` backup, version bump, ROI injection; `_can_update_local_skill_metadata()` path guard.

### 14.2 Integration Tests

- **A/B runner**: Mock agent CLI returning known JSON; timeout enforcement with sleeping mock; concurrency verification; error propagation through `asyncio.gather()`.
- **Viewer HTTP**: `POST /feedback` writes `feedback.json`; `GET /` serves HTML with correct metadata; concurrent POSTs handled safely.
- **Skill helper lifecycle**: `list-skills` ŌåÆ `resolve-skill` ŌåÆ `accept-skill` ŌåÆ `store-skill` end-to-end; ambiguous prefix handling; local-mode server skip.
- **Aggregator**: Known eval-full data produces expected benchmark; length mismatch detection.

### 14.3 Orchestrator Tests

- Full SKILL.md end-to-end with mock skill against mock A/B server; all 8 phases execute in order.
- Variable propagation: `MEGA_DIR`, `ITER_DIR`, `ITERATION_NUM`, `SKILL_NAME`, `SKILL_PATH`, `EVAL_AGENT` consistent across Bash blocks.
- Plugin cache path filtering: `CLAUDE_PROJECT_DIR` under cache ŌåÆ empty `PROJECT_DIR_ARG`.
- Edge cases: no skills found, single skill auto-selection, canonical directory already correct.

### 14.4 Security Tests

- Pattern matching: all 10 patterns with edge cases (multi-line, comment vs. code block).
- Trust classification: author marker variants (case sensitivity, nested vs. top-level).
- Policy derivation: exhaustive `(trust_level, findings) ŌåÆ policy` truth table.
- Workspace path traversal: `../` in artifact names, empty skill names, null bytes.

## 15. Known Limitations and Future Considerations

### Current Limitations

1. **Single-model A/B testing**: Both arms use the same agent CLI. No cross-model comparison without separate invocations.
2. **Fixed 4 test cases**: Convention-driven, not schema-enforced. `aggregate_eval_result()` accepts any count, silently changing statistical weight.
3. **No streaming progress during A/B execution**: The user sees no output for up to `MEGA_CODE_EVAL_TIMEOUT` seconds while completions run.
4. **Port 3117 hardcoded**: Both viewer scripts and `enhancement_viewer.py` default to 3117. `launch-viewer.sh` kills any existing process on that port, potentially terminating unrelated processes.
5. **Token estimation fallback**: When agent CLI omits token counts, `len(output_text)` is substituted ŌĆö not a real token count. `token_savings` may be unreliable.
6. **No cross-iteration trend analysis**: Each iteration is independent. `load_previous_iteration()` exists but is not called by the orchestrator. No automated trend or regression detection.
7. **No regression guard**: A NOT_BENEFICIAL verdict in iteration N can silently replace a better skill from iteration N-1. The user must catch this manually.
8. **Synchronous grading by host agent**: Grading uses the same LLM as the orchestrator. No independent grader, no cross-validation, no parallel grading.
9. **Per-completion timeout, not per-phase**: With lower concurrency, total Phase 4 time grows proportionally. No global phase timeout.
10. **No re-authentication during store**: If `MEGA_CODE_API_KEY` expires between auth check and Phase 8, the upload fails with local-only save. No retry prompt.

### Future Considerations

1. **Cross-model evaluation**: `--eval-model` flag for comparing skill effect across different model capabilities.
2. **Configurable viewer port**: `MEGA_CODE_VIEWER_PORT` env var to prevent port conflicts.
3. **Streaming A/B progress**: Progress callbacks in `_run_ab()` emitting per-test completion events to stderr.
4. **Automated regression guard**: Compute whether the new verdict represents a regression before accepting. Require explicit confirmation.
5. **Cross-iteration trend visualization**: Call `load_previous_iteration()` from the orchestrator and display delta metrics.
6. **Independent grading**: Route grading through a separate `host_llm.complete()` call or configurable grader model.
7. **Grading schema enforcement**: Pydantic validation for grading JSON with recovery prompting.
8. **Streaming token count**: Protocol with `claude` CLI for per-token streaming counts.

## Appendix A: Glossary

| Term | Definition |
|------|------------|
| **A/B Testing** | Evaluation methodology: each test case executed twice (with-skill and baseline) in isolated subprocess completions |
| **Baseline** | A/B arm without system prompt. Minimal directive prevents tool use. |
| **With-Skill** | A/B arm with full SKILL.md content as `system_prompt` |
| **Expectation** | Natural-language assertion in `EvalTestCase` about expected output quality. Typically 3ŌĆō4 per test case. |
| **Grading** | Evaluating whether a specific expectation was met. Produces `EvalGrading(expectation, passed, evidence)`. |
| **Verdict** | Binary outcome: `BENEFICIAL` or `NOT_BENEFICIAL`. Based on configurable performance lift and token savings thresholds. |
| **BENEFICIAL** | Positive verdict: skill demonstrably improves quality (>5% lift) or reduces verbosity (>20% token savings without degradation). |
| **ROI** | Return on Investment: `performance_increase` + `token_savings`. Injected into enhanced skill frontmatter as `metadata.roi`. |
| **Trust Level** | Provenance classification: `trusted` (mega-code authored) or `semitrusted` (unknown/third-party). Gates `skip_ab` policy. |
| **Security Score** | Discrete integer from {1, 3, 4, 5} ŌĆö defined in FR-4.4. |
| **Iteration** | Single complete pass through Phases 1ŌĆō8. Each creates `iteration-N/` in the workspace. |
| **Enhancement Principles** | Four rules for Phase 7: generalize, stay lean, explain reasoning, bundle repeated work. |
